# Работа с GitHub

## Создание SSH ключа
Для работы с GitHub небходимо создать ssh ключ

```bash
ssh-keygen # Your public key has been saved in /home/user/.ssh/id_rsa.pub
```

Теперь надо добавить его на GitHub
Чтобы посмотреть содержимое ключа
```bash
cat /home/user/.ssh/id_rsa.pub      # Linux / WSL

type /c/Users/yutya/.ssh/id_rsa.pub # Windows
```

Содержимое ключа нужно добавить на гитхаб

1. Нажимаем на свою аватарку справа сверху -> Settings
2. Переходим во вкладку SSH and GPG keys
3. Нажимаем New SSH key
4. В поле key вводим полученный ключ и сохраняем
5. Готово!

Чтобы проверить работает ли ключ:

```bash
ssh -T git@github.com # Hi FreshinS! You've successfully authenticated, but GitHub does not provide shell access.
```

## Работа с репозиториями

Для того чтобы склонировать репозиторий на GitHub справа сверху жмём на зелёную кнопку "<> Code", обязательно выбираем SSH и копируем. Затем в терминале пишем

```bash
git clone git@github.com:HexDiplom/diplom-front.git # после clone - url которое вы скопировали
```

Склонировав репозиторий переходим в основую ветку разработки dev

```bash
git switch dev
```

В наших репозиториях ветки main и dev запротекчены, то есть напрямую в них запушить нельзя. Для пуша в эти ветки необходимо создания Pull Request из личной ветки в dev (об этом ниже), и дальнейшее получение аппрува от меня.

```
main - основная ветка с готовым протестированным функционалом без багов и лишнего кода
dev - основная ветка разработки, сюда попадает только готовый, но ещё не протестированный функционал, здесь могут быть баги, но код обязан работать и запускатся
```

Для того чтобы начать работать необходимо создать свою ветку (Желательно использовать понятные названия веток описывающие какой функционал будет реализован, или какие проблемы будут исправлены, например - "feat-authorization", "fix-validation" и т.д.)

```bash
git switch -c <Название_Ветки>
```

Сделав определённые изменения не забываем делать коммиты, их тоже желательно понятно называть. Не нужно коммитить каждую строчку кода, но и заливать 5 тыс. строчек одним коммитом не желательно, соблюдайте баланс :D
```bash
git add <путь_до_файла>            # Добавить файл в индекс (то что полетит в коммит)
git add .                          # Добавить все файлы в индекс (не желательно)
git status                         # Проверить какие файлы попадут в коммит (Не нужно пожалуйста пушить .env, добавьте его в .gitignore)
git restore <путь_до_файла>        # Убрать лишний файл из индекса
git commit -m "Название_Коммита"   # Сам коммит
```

Закончив работу на сегодня, или доделав определённый функционал пушим в репозиторий
```bash
# Конфигурация Git если ещё не сделана
git config --global user.name "Ваш ник GitHub или имя"
git config --global user.email "Ваш email"

# Сам пуш
git push

git push -u origin HEAD # Можно попробывать если с обычным пушем ошибка

git push --force # ТАК НИКОГДА НЕ ПИШЕМ, ХОТЯ У ВАС И НЕ ПОЛУЧИТСЯ, НО ПРОБЫВАТЬ НЕ НАДО
```

Как только вся работа в ветке завершена, функционал протестирован, все пайплайны пройдены, можно создавать Pull Request.

1. Заходим в репозиторий -> Pull Requests -> New Pull Request
2. base - ветка в которую будут вливаться изменения, выбираем dev
3. compare - ветка из которой берём изменения, выбираем свою
4. Даём понятное название PR описывающее сделанные изменения во всех коммитах
5. Ждём аппрув от меня
6. Жмём Squash and merge
